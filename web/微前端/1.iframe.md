# 为什么要用微前端

1.   团队规模大，开发人员多的情况
2.   架构复杂，多个系统，项目可组合的情况,
3.   技术栈多样

不需要使用微前端的情况

1.   **项目并不复杂**，没必要拆分成多模块
2.   项目复杂，但是**开发人员不多**的情况，也没必要使用微前端，微前端会极大增加项目的复杂度
3.   微前端对首屏加载不友好，微前端首次加载通常更慢

# 微前端

微前端`（Micro Frontend）`是将前端应用拆分成多个独立的、可部署的子应用，每个子应用由不同的团队独立开发、部署和维护，但最终在同一个页面中协同工作。

目标：

1. 降低大型前端系统的复杂度。
2. 支持多团队并行开发。
3. 各子应用技术栈独立，可渐进式升级。

**常见实现方式：**

- **`iframe`**（最早方案）
- **JavaScript 沙箱 / 微应用框架**（如 `Qiankun`、`Single-SPA`）
- **Web Components**（标准化自定义组件）

# `iframe` 实现微前端

`iframe` 是 HTML 标签 `<iframe>`，可以在页面中嵌套另一个完整的网页。
 最早的微前端方案就是用 `iframe` 来隔离子应用。

`iframe` 实现微前端简单、安全、隔离性好，但性能、通信、样式和路由整合差，因此在现代微前端中，`iframe` 更常用于**完全独立、跨域的嵌套场景**，而对于同域、多子应用协作的场景，更多使用 **JS 沙箱 + 微应用框架** 来替代 `iframe`。

### 1. 优点

1.  **隔离性强**
    -   `iframe` 有独立的浏览器上下文（独立的 DOM、CSS、JS 作用域），不同子应用不会互相污染。
    -   避免全局变量冲突，尤其是第三方库版本不一致的问题。
2.  **部署独立**
    -   每个子应用可以独立部署，URL 可以是完全不同的服务。
    -   子应用崩溃不会直接影响主应用。
3.  **安全性高**
    -   `iframe` 可以通过 `sandbox` 属性控制权限，限制子应用操作主页面或访问敏感数据。
4.  **简单直接**
    -   使用 `iframe` 只需要 `<iframe src="子应用地址">`，不需要复杂的框架支持。

------

### 2. 缺点

1.  **性能开销大**
    -   每个 `iframe` 都是一个完整浏览器环境，加载、渲染、JS 执行都独立，占用资源多。
    -   大量 `iframe` 会导致页面卡顿。
2.  **样式和布局不灵活**
    -   `iframe` 的内容在独立文档里，父页面无法直接控制内部样式。
    -   响应式布局、统一样式管理困难。
3.  **通信麻烦**
    -   父子应用之间只能通过 `postMessage` 进行跨域通信。
    -   数据共享和事件传递不够自然，需要写桥接代码。
4.  **路由整合困难**
    -   `iframe` 内部有独立路由，父应用的路由无法直接影响子应用，页面跳转体验差。
5.  **SEO 和首屏加载不友好**
    -   `iframe` 内的内容搜索引擎难以索引。
    -   首屏加载时可能出现空白，影响用户体验。
6.  **调试困难**
    -   `iframe` 内部 JS 调试、性能分析不如单页面应用方便。
